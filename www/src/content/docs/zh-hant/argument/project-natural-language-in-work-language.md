專案中承載業務語義的自然語言部分，應與工作語言一致。

工作語言，指專案在理解、命名、判斷、記錄和維護業務語義時實際使用，並且能夠被主要維護者持續複用的自然語言。它由業務概念、業務規則、客戶口徑、產品取捨和日常記錄中的穩定用詞共同形成；可以出現在團隊協作中，也可以出現在獨立開發者的個人工作中。

## 背景

專案真相中有一部分由程式碼、測試、設定等實作載體表達；另一部分由自然語言、術語表、決策紀錄和必要的歷史記憶補足。本文討論的是後一類內容中承載業務語義的自然語言表達，也包括程式碼註解中解釋業務含義、領域規則和產品取捨的內容。

這類自然語言承擔的是理解和判斷任務。它記錄業務概念如何命名，規則如何成立，客戶口徑如何解釋，產品取捨如何被持續維護。它的語言選擇會直接影響人、LLM 和 agent 如何檢索、理解、更新專案。

## 理由

專案自然語言部分應與工作語言一致，因為工作語言最貼近專案實際處理業務語義的方式。

如果團隊、客戶和業務現場使用中文術語討論業務，卻長期用英文記錄自然語言專案真相，專案就會持續經歷轉譯。LLM 可以理解多種語言，但跨語言轉寫仍會帶來近似翻譯、術語漂移、表達分散和檢索斷裂。同一業務概念在業務現場、術語表、需求說明、程式碼註解和測試說明中出現多套說法，後續維護者就需要反覆判斷它們是否指向同一件事。

工作語言一致，可以讓業務概念在專案中保持穩定。術語表可以圍繞真實使用的詞建立；全文搜尋、腳本檢索、上下文抽取和投影視圖生成，也能更容易命中目標內容。人、LLM 和 agent 進入專案時，可以直接接觸專案實際使用的業務語義，減少對跨語言近似表達的依賴。

實作命名和自然語言命名可以各自穩定。專案中可能存在某個英文元件名、函式名或資料表名，用來表達目前程式結構；自然語言中的業務概念則可以使用工作語言中的穩定說法。專案重構後，實作名可能變化，業務概念仍然可以保持穩定。實作命名記錄實作結構，自然語言記錄業務語義，二者透過術語表、引用關係和上下文建立對應。

因此，自然語言部分與工作語言一致，可以讓業務語義保持穩定，也能讓實作結構按工程需要演進。

## 邊界

本文討論的是專案中承載業務語義的自然語言部分。判斷一段文字是否適用本文原則，關鍵看它是否在幫助專案理解概念、說明規則、確認口徑和記錄取捨。它可以出現在文件、術語表、測試說明或程式碼註解中；載體不同，語言原則相同。

實作符號按工程約束處理。程式、資料、介面和第三方系統命名，可以遵循相應生態與團隊慣例。關鍵要求是同一業務概念在整個系統中保持穩定對應，避免自然語言說明和實作別名之間分散成無法匹配的多套說法。

主要服務技術維護的說明，可以按工程協作習慣處理。它們關注工程操作、框架限制、建置流程和實作細節，語言選擇由團隊工程實踐決定。

## 反對意見

**各專案都使用自己的工作語言，會不會降低跨專案複用？**

跨專案複用重點在語義遷移。LLM 可以讀取來源專案中的領域知識、規則模式和術語關係，再結合目標專案的術語表、既有專案真相、程式碼結構和工作語言，生成符合目標專案表達習慣的版本。

這樣遷移後的內容，能夠像為目標專案原生編寫的一樣參與檢索、審查和持續維護。能夠遷移的是領域知識、規則模式、建模經驗、術語關係和歷史決策的啟發；只有被目標專案確認並採用的內容，才會成為目標專案真相。

**自然語言與實作命名不同，會不會製造混亂？**

實作命名和自然語言命名表達專案真相的不同部分。實作命名表達程式結構，受元件劃分、重構、框架慣例、資料庫設計和符號系統影響；自然語言命名表達業務語義，需要跟隨工作語言保持穩定。

專案重構後，實作名可能變化，業務概念仍然可以保持穩定。二者透過術語表、引用關係和上下文建立對應，能夠同時支援工程定位和業務理解。

## 結論

專案自然語言部分應與工作語言一致。工作語言承載專案實際處理業務語義的方式，也能被主要維護者持續複用。

這條原則讓專案真相中的自然語言內容、術語表、業務規則、驗收口徑、決策紀錄和業務含義註解保持同一語義空間，減少轉譯損耗和術語漂移，提高檢索、抽取和投影視圖生成效率。實作命名可以按工程約束演進，自然語言部分則應穩定承載業務語義。
