在 coding agent 參與長期迭代的軟體專案中，專案真相必須進入程式碼庫，並與實作共享版本邊界。

本文沿用 [什麼是專案真相](/zh-hant/argument/what-is-project-truth/) 的定義：專案真相是目前專案實際依照的內容。其中一部分存在於程式碼實作中，另一部分由自然語言、術語表、決策紀錄和必要的歷史記憶補足。

已經由程式碼表達的部分，本來就在程式碼庫中。本文所說的「進入程式碼庫」，重點指需要由自然語言、術語表、決策紀錄和必要的歷史記憶補足的那部分專案真相；它們也要與程式碼實作處在同一個版本化空間裡。

## 背景

傳統產品與研發協作把需求和實作分開。產品團隊透過 PRD、Spec、Issue 等形式交付需求，研發再把這些描述轉化為程式碼。這套模式主要服務於人與人的非同步協作：不同角色、不同工具、不同交付邊界，需要一份可以轉交、審查和排期的材料。

這種分離來自組織協作結構，並不表示需求材料和實作必須長期分屬兩個版本空間。在 coding agent 參與實作之後，需求澄清和程式碼變更的距離被顯著縮短。很多時候，只要需求被明確定義，約束被澄清，上下文被補足，實作就可以很快生成。

需求材料的價值在於約束和指導實作。長期迭代專案需要把目前專案實際依照的內容保存在實作所在的版本系統中，避免它們滯留在外部文件、人的記憶、會議紀錄或過期 issue 裡。

## 理由

程式碼沒有表達、難以表達或容易被誤讀的專案真相停留在程式碼庫之外，會製造持續的轉譯成本。

傳統 PRD、Spec、Issue 往往先從既有專案狀態中抽取上下文，再加入背景、功能描述、變更說明和驗收標準。這一步會把程式碼實作、產品現狀和業務判斷轉譯成文件。研發或 agent 接收到這些材料後，還要把文件重新映射回程式碼庫，判斷它對應哪些模型、介面、頁面、流程、邊界條件和測試。於是，同一份專案真相經歷了從程式碼到文件、再從文件回到程式碼的往返轉譯。

每一次轉譯都會帶來資訊損失、轉述偏差、版本不同步和上下文缺失。在人類協作主導的模式下，這種成本可以被組織流程吸收。到了 agent 參與實作時，這些轉譯會持續阻塞程式碼變更。agent 主要在程式碼庫中工作，實際變更最終表現為程式碼 diff，並透過測試、建置和部署驗證。程式碼沒有表達、難以表達或容易被誤讀的專案真相一旦停留在另一個版本空間，agent 每次改動都要依賴使用者轉述、臨時檢索或額外拼接上下文。

專案真相進入程式碼庫後，這段跨版本空間的往返轉譯會被消除。程式碼實作繼續表達專案真相中可以執行的部分；自然語言、術語表、決策紀錄和必要的歷史記憶，在同一個版本邊界內補足程式碼沒有表達、難以表達或容易被誤讀的部分。兩部分一起維護後，變更可以直接發生在實作所在的版本系統中，而不必先生成一份外部文件，再把文件翻譯回程式碼變更。

外部知識庫仍然有價值，適合閱讀、討論、展示和收集材料。它的弱點在於版本邊界：它不與程式碼共享提交歷史、審查流程、回滾邊界和變更上下文。對於長期迭代專案來說，目前專案實際依照的內容如果不隨實作同步演進，就會逐漸變成不可靠的旁路資訊。

## 邊界

進入程式碼庫的是已經成為專案真相的內容；產生這些內容的過程，可以留在各自合適的工作流中。

團隊會議、使用者回饋、產品討論、與 coding agent 的多輪澄清、基於回饋自動觸發的 pull request，都可以產生專案真相。這些過程本身不必原樣進入程式碼庫。只有那些經過澄清、壓縮和確認，並成為目前專案實際依照的內容，才需要沉澱為目前版本的業務狀態、領域約束、真實世界條件、商業約束、產品語義、穩定術語、決策結果和關鍵變化原因。

這套模式主要適用於需要長期迭代的軟體專案。長期迭代需要穩固、可追溯、可複用的目前依據。單次原型探索的目標是快速做出來、快速看見、快速回饋。此時專案真相可以暫時存在於對話上下文、臨時材料和即時判斷中；過早沉澱進程式碼庫，反而會拖慢探索速度。

專案真相進入程式碼庫，是底層結構判斷；上層如何存取、審計、修改和發布這些真相，屬於應用層設計問題。產品、設計、業務和研發不必都直接面對 Git 或程式碼編輯器。GUI、對話式 agent、需求系統或自動化工作流，都可以作為審計、核查和更新專案真相的入口。

決定專案真相的位置時，應先看產品如何被定義、實作、驗證和迭代，再看不同人員、agent 和工具應該透過哪些入口創造價值。

## 反對意見

**程式碼庫會不會變成產品材料垃圾場？**

這個風險來自入庫標準失守。若是把所有與產品有關的材料都放進倉庫，程式碼庫必然會失控。

本文採用更窄的入庫標準：進入程式碼庫的內容必須已經成為目前專案實際依照的內容。使用者研究、市場調研、商業分析可以作為輸入留在獨立專案中。只有當這些材料的結論已經轉化為產品行為、權限規則、領域模型、真實世界限制、商業約束、部署限制、穩定術語或方案取捨時，才需要隨實作版本化。

**PRD、Spec、Issue 是否因此被否定？**

不會。PRD、Spec、Issue 的名稱或形式本身沒有問題；問題出在傳統用法把它們放在需求與實作之間，承擔最終依據的角色。

在 agent 時代，它們可以作為上層封裝繼續存在：討論入口、任務編排、審計介面、變更申請或歷史紀錄。要進入程式碼庫中的專案真相，則需要經過澄清、壓縮、確認和轉化。

## 結論

在 coding agent 參與長期迭代的專案中，不能繼續把程式碼沒有表達、難以表達或容易被誤讀的專案真相放在實作所在的版本系統之外。需求和實作本來服務於同一個目標；coding agent 縮短了澄清到變更的距離，也讓傳統分離的成本更明顯。

專案真相必須進入程式碼庫，因為目前專案實際依照的內容需要與實作共享同一個版本邊界。這樣，人、agent 和工具才能圍繞同一份目前依據工作，減少在外部材料和程式碼變更之間來回搬運上下文。

本文只確定位置與版本關係，不展開專案真相進入程式碼庫後的具體形式、審計流程、更新機制，以及它與測試、型別、設定、文件和 agent 工作流的協同方式。
