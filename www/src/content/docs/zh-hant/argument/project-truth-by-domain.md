本文沿用 [專案真相必須進入程式碼庫](/zh-hant/argument/project-truth-in-codebase/) 的判斷：程式碼沒有表達、難以表達或容易被誤讀的專案真相，也要與實作共享版本邊界。本文將這部分專案真相簡稱為非程式碼專案真相，並進一步討論它們在程式碼庫中應如何組織。

專案真相應按領域組織。領域，是專案真相的組織單位，用來把同一範圍內需要一起理解、修改、審查和持續維護的程式碼實作、術語、約束、規則、決策紀錄和必要的歷史記憶組織起來；這些內容共同構成該領域目前有效的專案真相。

領域組織的優先形態，是貼近實作的目錄結構。目錄不僅提供放置位置，也透過層級關係形成背景疊加：上級領域沉澱較大範圍內成立的專案真相，下級領域沉澱本領域範圍內更具體的專案真相。程式碼庫中良好的領域劃分，能夠讓專案真相更容易被發現、理解、審查和同步維護。程式碼目錄無法表達領域邊界時，可以透過外置領域聲明界定領域範圍。

## 背景

在程式碼庫中組織專案真相時，還需要回答如何放置、如何發現、如何維護的問題。既有程式碼表達了專案真相中可以執行的部分；自然語言、術語表、決策紀錄和必要的歷史記憶補足程式碼無法表達、難以表達或容易被誤讀的部分。這些非程式碼專案真相需要與對應實作形成穩定關係。

如果缺少組織方式，非程式碼專案真相即使與程式碼共享版本邊界，也會退化成材料堆積。人和 agent 需要知道某條真相適用於哪部分實作，修改程式碼時應該查看哪些背景，審查變更時又應該同步更新哪些非程式碼專案真相。

因此，專案真相還需要一種貼近專案理解和修改邊界的組織方式。

## 理由

專案真相應按領域組織，因為領域是理解和修改專案時需要共同背景的單位。

領域邊界來自專案中穩定的理解和維護邊界。它可以來自業務能力，也可以來自技術架構、元件體系、基礎設施或實作邊界。判斷標準是：這些內容是否需要被一起理解、修改、審查和持續維護。

當領域邊界與程式碼目錄邊界一致時，該領域的非程式碼專案真相應放在領域實作旁。這種放置方式心智負擔低，也更容易被發現。開發者或 agent 進入某個目錄處理問題時，可以直接看到該領域目前有效的術語、約束、規則、決策紀錄和必要的歷史記憶；專案真相的閱讀、修改和審查，也能跟隨程式碼變更發生在同一個版本邊界內。

目錄結構還會形成背景疊加。上級領域沉澱較大範圍內成立的專案真相，下級領域沉澱本領域範圍內更具體的專案真相。這種背景疊加來自程式碼實作的自然結構，用來呈現不同層級的關注範圍，不製造強制覆蓋規則。它讓局部真相能夠放回更大的實作和領域背景中理解，降低反覆解釋上下文的成本。

良好的程式碼領域劃分，會顯著降低專案真相架構的實施成本。既有程式碼結構能夠表達領域邊界時，應優先複用這種結構。非程式碼專案真相貼近領域實作，專案真相就更容易保持目前有效，也更容易在修改中被同步維護。

## 邊界

專案真相按領域組織，目錄是領域邊界最常見、最自然的載體。一個目錄是否構成領域單位，需要透過約定入口界定。這個入口用於讓人和 agent 判斷該目錄是否承載非程式碼專案真相，以及這些真相適用於哪個領域範圍。

並非所有領域都天然對應目錄。有些程式碼結構以檔案集合表達領域，例如同一目錄下混合多個模組。此時可以透過外置領域聲明界定領域範圍，並把非程式碼專案真相綁定回對應路徑和明確檔案清單。領域單位仍然按領域成立，路徑和檔案範圍只是它在程式碼庫中的定位方式。

檔案範圍領域應保持邊界可枚舉、可審查、可穩定維護。明確檔案清單是這類領域的表達上限。一個領域如果需要依賴 include、exclude、glob 等複雜規則才能描述，說明它已經超出離散檔案集合適合承擔的複雜度，應當考慮透過目錄結構重新組織為更清晰的領域單位。

定義了檔案範圍的領域，是最終領域單位。它不依賴目錄層級提供下級空間，也不繼續產生下級領域關係。在檔案集合內部繼續劃分子領域，會削弱邊界的直觀性和維護性。

專案真相按領域組織，不要求專案為了記錄專案真相而立即重排現有程式碼結構；但程式碼庫中清晰的領域劃分，是實施專案真相架構的更優基礎。既有結構不能充分表達領域邊界時，可以用外置領域聲明補足；當某個領域長期只能依賴複雜檔案規則才能界定時，通常說明程式碼結構中的領域抽象不足。

## 反對意見

**領域知識的結構更接近圖，為什麼不用知識圖譜？**

領域之間確實會形成圖狀關係。一個領域可能依賴另一個領域的術語、呼叫另一個領域的介面、受到另一個領域的業務規則影響，也可能與多個領域共享同一段歷史背景。

知識圖譜適合關係數量大、關係類型多、需要程式化探索的知識場景，例如搜尋、推薦、大型企業知識管理和跨系統知識分析。專案真相的主要場景更具體：它服務於一個程式碼庫內的理解、修改、審查和持續迭代。這個場景已經有目錄、檔案路徑、程式碼依賴、術語表、連結和全文搜尋作為基礎設施，領域之間的關係通常可以透過這些線索發現。

專案真相的主維護結構應優先使用程式碼庫原生結構。把非程式碼專案真相放在領域實作附近，可以讓發現、閱讀、修改和審查發生在同一個工作流中。知識圖譜引入另一套關係建模和維護介面，在程式碼庫層級通常成本高於收益。

知識圖譜可以用於生成投影視圖。它可以從專案真相中抽取、重組和呈現關係，用於跨領域搜尋、關係瀏覽、審計或分析。專案真相本身就是目前依據，圖狀結構更適合作為基於專案真相生成的投影視圖。

**有些領域並不天然以目錄切分，怎麼辦？**

專案真相的組織單位是領域，目錄只是領域邊界的一種自然形態。

當程式碼目錄已經表達領域邊界時，專案真相應就近放在該目錄中。當領域邊界散落在同一目錄的多個檔案中時，可以用外置領域聲明界定該領域，並透過路徑和明確檔案清單綁定到對應實作。

這種方式保留了領域組織原則，也避免把目錄誤當成唯一組織形態。外置領域聲明用於補足程式碼結構沒有表達清楚的領域邊界。它適合處理少量、明確、可枚舉的檔案範圍；當邊界複雜到需要規則系統維護時，更好的方向是整理程式碼結構中的領域抽象。

**這裡的領域和 DDD 的領域有什麼區別？**

本文中的領域用於組織專案真相。判斷一個範圍是否構成領域，要看其中的程式碼實作和非程式碼專案真相是否需要一起理解、修改、審查和持續維護。

DDD 中的領域服務於業務建模和系統設計，關注業務問題、模型語言、限界上下文和實作邊界。本文中的領域服務於專案真相組織，因此範圍可能覆蓋 DDD 語境之外的介面層、應用層、元件體系、基礎設施或其他技術結構。

如果專案採用 DDD，它形成的業務邊界可以成為專案真相領域劃分的重要依據。本文不要求專案採用 DDD，也不把領域限定為 DDD 語境中的業務領域。

## 結論

在程式碼庫中，專案真相應按領域組織。領域把需要共同理解、修改、審查和持續維護的程式碼實作與非程式碼專案真相放在同一個語義範圍內，使目前依據能夠隨實作一起維護。

目錄結構是領域組織的優先形態。它心智負擔低、貼近實作、易於發現，並能複用專案技術架構中已經形成的領域抽象。領域邊界與目錄邊界一致時，應就近放置；領域邊界無法由目錄表達時，可以透過外置領域聲明綁定到明確路徑和檔案範圍。

按領域組織專案真相，可以保留程式碼庫原生結構的可理解性，也為後續編譯、掃描和生成投影視圖提供穩定基礎。知識圖譜、跨領域關係瀏覽和審計分析，可以透過投影視圖呈現；專案真相的主維護結構，應由領域組織承載。
