專案真相，是目前專案實際依照的內容。它決定專案現在如何執行、如何被理解、後續修改和判斷時必須參考什麼，也為不同任務場景生成投影視圖提供依據。

專案真相主要服務於專案的持續迭代。只要專案會被繼續理解、修改、解釋、審計，或被用來生成投影視圖，就需要一份目前依據。使用這份依據的可以是人，也可以是 agent；專案真相關注的是持續變化中的理解和判斷。

## 背景

使用專案真相這個說法，是為了強調目前有效。它指目前版本中專案實際依照的核心內容；這些內容可以隨版本變化，其範圍由專案是否採用，以及目前是否有效來限定。

文件、上下文、事實分別指向不同問題：文件強調表達形式；上下文通常和場景綁定，更像某次任務或某個視角下的切面；事實是關於領域或真實世界的客觀陳述。專案真相關心的是專案從這些材料中實際採用了什麼，並按什麼成立、執行和演進。

因此，定義專案真相時，核心問題是：目前專案實際依照什麼？哪些內容一旦變化，會影響專案的執行、理解、維護、判斷或投影視圖生成？

## 理由

**專案真相的一部分，就存在於程式碼實作中。**

目前實作已經表達了產品實際怎樣執行、資料如何組織、規則如何落地、邊界如何處理。程式碼已經清楚表達的真相，保留在程式碼裡即可，不要再用文件複寫同一份真相。複寫會製造同步成本，也會增加理解難度。

**自然語言更適合補足程式碼沒有表達、難以表達，或容易被誤讀的部分。**

這些內容包括程式碼無法充分表達的領域知識、隱性知識、真實世界知識、業務規則、商業約束和實際生效的產品權衡。例如，領域中確實存在某些事實，但專案真正需要記錄的是自己如何採用這些事實，並如何把它們落到產品行為和實作方式裡。

「人有恆牙和乳牙」是領域事實；「本系統採用 FDI 編號記錄牙齒，早期未區分恆牙乳牙，現在統一用恆牙編號記錄，並增加布林變數區分恆牙乳牙」才是專案真相。前者是領域知識，後者是目前系統採用的記錄規則和歷史取捨。

**穩定術語表也屬於專案真相。**

專案需要在同一套詞彙中被理解和修改。術語表的作用是消除模糊空間，減少同義詞、近義詞、業界俗稱和錯誤寫法帶來的歧義。在 agent coding 中，這一點尤其重要：agent 會根據詞彙建立上下文關聯。如果專案沒有穩定術語表，就容易混用「帳號」和「帳戶」、「使用者」和「客戶」，甚至把「正頜」和「正顎」這類專業詞弄錯。

統一詞彙的價值在於讓人和 agent 在同一個語義空間裡工作。

**專案真相還需要保留少量歷史記憶。**

這裡的歷史用於記錄關鍵變化原因，完整討論過程可以留在提交紀錄、issue 或外部材料裡。平時閱讀專案時，可以先看目前仍然生效的決策、規則和權衡；這些內容應該直接體現在目前專案真相裡。

歷史的作用是在需要追問「為什麼曾經改成這樣」時，提供低成本的時間維度線索。Git 通常能回答改了什麼、什麼時候改的、誰改的；專案真相中的歷史記錄主要回答為什麼這樣改。因此，它適合記錄領域策略、業務邏輯、產品權衡、對外交付語義、重要術語等關鍵變化的原因。

這類歷史記錄要謹慎寫入，也要謹慎處理。寫入之後，刪除或壓縮需要明確專案判斷；單個 agent 當下看不出用途，不足以成為清理理由。未來的追溯、審計、投影生成或對外交付解釋，可能正需要這些變化原因。

## 邊界

**領域事實可以成為專案真相的來源。**

這裡說的事實，指關於領域或真實世界的客觀陳述。一個事實只有被目前專案採用，並且影響產品行為、業務解釋、實作判斷、術語使用、對外交付或後續維護時，才成為專案真相。

**多種載體都可以表達專案真相。**

程式碼、測試、設定、自然語言文件、術語表和決策紀錄，都可以表達專案真相。程式碼已經清楚表達的部分，可以直接保留在程式碼裡；文件主要補足程式碼沒有表達、難以表達或不適合承擔的部分。

**專案真相的範圍由專案需要劃定。**

開發者、產品經理、設計師、領域專家或業務方掌握的內容，都可能貢獻專案真相。真正進入專案真相的，是目前專案需要被持續複用的部分，範圍可能跨越實作、領域、業務、商業、產品和真實世界限制。

**投影視圖是專案真相的消費結果。**

投影視圖，是針對某次任務、某個 agent 執行場景、某次審計、業務理解、產品調研或管理檢視，從專案真相中抽取、重組並呈現一部分內容。它可以被生成、展示、快取、傳遞給 agent 使用。只要能夠基於專案真相語義重現，通常保持為生成結果或快取結果即可。

本文只定義專案真相是什麼。專案真相如何進入版本庫，如何審查、更新、廢棄和治理，不在本文展開。

## 反對意見

**專案真相會不會變成什麼都往裡放？**

這個擔心成立。專案真相一旦缺少治理，就會退化成材料堆積。因此，專案真相必須有治理機制：進入有標準，變更有審查，並且隨版本庫一起維護。

進入專案真相的內容，應當是目前專案實際依照的內容。目前版本已經停止採用的內容，應當隨專案變更一起修改並提交，避免繼續混在目前真相中。

**既然專案真相會隨版本變化，為什麼還叫真相？**

這裡的真相強調目前有效，並允許隨版本變化。

某個業務規則、術語、實作方式或產品權衡，可能曾經是專案真相，也可能在新版本中被替換。這種變化符合軟體專案的版本化存在方式。

對任一版本來說，專案實際依照什麼，人和 agent 就應該按什麼理解、修改和判斷專案。專案真相的意義，正在於明確這個目前版本的有效核心。

## 結論

專案真相，是目前專案實際依照的內容。一部分由程式碼實作表達，另一部分由自然語言、術語表、決策紀錄和必要的歷史記憶補足。

它關注的是：目前專案如何執行，如何被理解，後續修改時必須尊重什麼，以及不同任務場景下的投影視圖可以從哪裡生成。事實材料可以作為輸入，投影視圖可以作為輸出；只有被目前專案採用並持續影響判斷的內容，才需要沉澱為專案真相。

先說明專案真相是什麼，專案真相架構才有明確的基礎。
