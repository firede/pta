本文沿用 [项目真相必须进入代码库](/argument/project-truth-in-codebase/) 的判断：代码没有表达、难以表达或容易被误读的项目真相，也要与实现共享版本边界。本文将这部分项目真相简称为非代码项目真相，并进一步讨论它们在代码库中应如何组织。

项目真相应按领域组织。领域，是项目真相的组织单位，用来把同一范围内需要一起理解、修改、审查和持续维护的代码实现、术语、约束、规则、决策记录和必要的历史记忆组织起来；这些内容共同构成该领域当前有效的项目真相。

领域组织的优先形态，是贴近实现的目录结构。目录不仅提供放置位置，也通过层级关系形成背景叠加：上级领域沉淀较大范围内成立的项目真相，下级领域沉淀本领域范围内更具体的项目真相。代码库中良好的领域划分，能够让项目真相更容易被发现、理解、审查和同步维护。代码目录无法表达领域边界时，可以通过外置领域声明界定领域范围。

## 背景

在代码库中组织项目真相时，还需要回答如何放置、如何发现、如何维护的问题。已有代码表达了项目真相中可以运行的部分；自然语言、术语表、决策记录和必要的历史记忆补足代码无法表达、难以表达或容易被误读的部分。这些非代码项目真相需要与对应实现形成稳定关系。

如果缺少组织方式，非代码项目真相即使与代码共享版本边界，也会退化成材料堆积。人和 agent 需要知道某条真相适用于哪部分实现，修改代码时应该查看哪些背景，审查变更时又应该同步更新哪些非代码项目真相。

因此，项目真相还需要一种贴近项目理解和修改边界的组织方式。

## 理由

项目真相应按领域组织，因为领域是理解和修改项目时需要共同背景的单位。

领域边界来自项目中稳定的理解和维护边界。它可以来自业务能力，也可以来自技术架构、组件体系、基础设施或实现边界。判断标准是：这些内容是否需要被一起理解、修改、审查和持续维护。

当领域边界与代码目录边界一致时，该领域的非代码项目真相应放在领域实现旁。这种放置方式心智负担低，也更容易被发现。开发者或 agent 进入某个目录处理问题时，可以直接看到该领域当前有效的术语、约束、规则、决策记录和必要的历史记忆；项目真相的阅读、修改和审查，也能跟随代码变更发生在同一个版本边界内。

目录结构还会形成背景叠加。上级领域沉淀较大范围内成立的项目真相，下级领域沉淀本领域范围内更具体的项目真相。这种背景叠加来自代码实现的自然结构，用来呈现不同层级的关注范围，不制造强制覆盖规则。它让局部真相能够放回更大的实现和领域背景中理解，降低反复解释上下文的成本。

良好的代码领域划分，会显著降低项目真相架构的实施成本。已有代码结构能够表达领域边界时，应优先复用这种结构。非代码项目真相贴近领域实现，项目真相就更容易保持当前有效，也更容易在修改中被同步维护。

## 边界

项目真相按领域组织，目录是领域边界最常见、最自然的载体。一个目录是否构成领域单位，需要通过约定入口界定。这个入口用于让人和 agent 判断该目录是否承载非代码项目真相，以及这些真相适用于哪个领域范围。

并非所有领域都天然对应目录。有些代码结构以文件集合表达领域，例如同一目录下混合多个模块。此时可以通过外置领域声明界定领域范围，并把非代码项目真相绑定回对应路径和明确文件列表。领域单位仍然按领域成立，路径和文件范围只是它在代码库中的定位方式。

文件范围领域应保持边界可枚举、可审查、可稳定维护。明确文件列表是这类领域的表达上限。一个领域如果需要依赖 include、exclude、glob 等复杂规则才能描述，说明它已经超过离散文件集合适合承担的复杂度，应当考虑通过目录结构重新组织为更清晰的领域单位。

定义了文件范围的领域，是最终领域单位。它不依赖目录层级提供下级空间，也不继续产生下级领域关系。在文件集合内部继续划分子领域，会削弱边界的直观性和维护性。

项目真相按领域组织，不要求项目为了记录项目真相而立即重排现有代码结构；但代码库中清晰的领域划分，是实施项目真相架构的更优基础。已有结构不能充分表达领域边界时，可以用外置领域声明补足；当某个领域长期只能依赖复杂文件规则才能界定时，通常说明代码结构中的领域抽象不足。

## 反对意见

**领域知识的结构更接近图，为什么不用知识图谱？**

领域之间确实会形成图状关系。一个领域可能依赖另一个领域的术语、调用另一个领域的接口、受到另一个领域的业务规则影响，也可能与多个领域共享同一段历史背景。

知识图谱适合关系数量大、关系类型多、需要程序化探索的知识场景，例如搜索、推荐、大型企业知识管理和跨系统知识分析。项目真相的主要场景更具体：它服务于一个代码库内的理解、修改、审查和持续迭代。这个场景已经有目录、文件路径、代码依赖、术语表、链接和全文搜索作为基础设施，领域之间的关系通常可以通过这些线索发现。

项目真相的主维护结构应优先使用代码库原生结构。把非代码项目真相放在领域实现附近，可以让发现、阅读、修改和审查发生在同一个工作流中。知识图谱引入另一套关系建模和维护界面，在代码库级别通常成本高于收益。

知识图谱可以用于生成投影视图。它可以从项目真相中抽取、重组和呈现关系，用于跨领域搜索、关系浏览、审计或分析。项目真相本身承担当前依据，图状结构更适合作为基于项目真相生成的投影视图。

**有些领域并不天然以目录切分，怎么办？**

项目真相的组织单位是领域，目录只是领域边界的一种自然形态。

当代码目录已经表达领域边界时，项目真相应就近放在该目录中。当领域边界散落在同一目录的多个文件中时，可以用外置领域声明界定该领域，并通过路径和明确文件列表绑定到对应实现。

这种方式保留了领域组织原则，也避免把目录误当成唯一组织形态。外置声明用于补足代码结构没有表达清楚的领域边界。它适合处理少量、明确、可枚举的文件范围；当边界复杂到需要规则系统维护时，更好的方向是整理代码结构中的领域抽象。

**这里的领域和 DDD 的领域有什么区别？**

本文中的领域用于组织项目真相。判断一个范围是否构成领域，要看其中的代码实现和非代码项目真相是否需要一起理解、修改、审查和持续维护。

DDD 中的领域服务于业务建模和系统设计，关注业务问题、模型语言、限界上下文和实现边界。本文中的领域服务于项目真相组织，因此范围可能覆盖 DDD 语境之外的接口层、应用层、组件体系、基础设施或其他技术结构。

如果项目采用 DDD，它形成的业务边界可以成为项目真相领域划分的重要依据。本文不要求项目采用 DDD，也不把领域限定为 DDD 语境中的业务领域。

## 结论

在代码库中，项目真相应按领域组织。领域把需要共同理解、修改、审查和持续维护的代码实现与非代码项目真相放在同一个语义范围内，使当前依据能够随实现一起维护。

目录结构是领域组织的优先形态。它心智负担低、贴近实现、易于发现，并能复用项目技术架构中已经形成的领域抽象。领域边界与目录边界一致时，应就近放置；领域边界无法由目录表达时，可以通过外置领域声明绑定到明确路径和文件范围。

按领域组织项目真相，可以保留代码库原生结构的可理解性，也为后续编译、扫描和生成投影视图提供稳定基础。知识图谱、跨领域关系浏览和审计分析，可以通过投影视图呈现；项目真相的主维护结构，应由领域组织承载。
