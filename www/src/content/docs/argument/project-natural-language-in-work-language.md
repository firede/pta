项目中承载业务语义的自然语言部分，应与工作语言一致。

工作语言，指项目在理解、命名、判断、记录和维护业务语义时实际使用，并且能够被主要维护者持续复用的自然语言。它由业务概念、业务规则、客户口径、产品取舍和日常记录中的稳定用词共同形成；可以出现在团队协作中，也可以出现在独立开发者的个人工作中。

## 背景

项目真相中有一部分由代码、测试、配置等实现载体表达；另一部分由自然语言、术语表、决策记录和必要的历史记忆补足。本文讨论的是后一类内容中承载业务语义的自然语言表达，也包括代码注释中解释业务含义、领域规则和产品取舍的内容。

这类自然语言承担的是理解和判断任务。它记录业务概念如何命名，规则如何成立，客户口径如何解释，产品取舍如何被持续维护。它的语言选择会直接影响人、LLM 和 agent 如何检索、理解、更新项目。

## 理由

项目自然语言部分应与工作语言一致，因为工作语言最贴近项目实际处理业务语义的方式。

如果团队、客户和业务现场使用中文术语讨论业务，却长期用英文记录自然语言项目真相，项目就会持续经历转译。LLM 可以理解多种语言，但跨语言转写仍会带来近似翻译、术语漂移、表达分散和检索断裂。同一业务概念在业务现场、术语表、需求说明、代码注释和测试说明中出现多套说法，后续维护者就需要反复判断它们是否指向同一件事。

工作语言一致，可以让业务概念在项目中保持稳定。术语表可以围绕真实使用的词建立；全文搜索、脚本检索、上下文抽取和投影视图生成，也能更容易命中目标内容。人、LLM 和 agent 进入项目时，可以直接接触项目实际使用的业务语义，减少对跨语言近似表达的依赖。

实现命名和自然语言命名可以各自稳定。项目中可能存在某个英文组件名、函数名或表名，用来表达当前程序结构；自然语言中的业务概念则可以使用工作语言中的稳定说法。项目重构后，实现名可能变化，业务概念仍然可以保持稳定。实现命名记录实现结构，自然语言记录业务语义，二者通过术语表、引用关系和上下文建立对应。

因此，自然语言部分与工作语言一致，可以让业务语义保持稳定，也能让实现结构按工程需要演进。

## 边界

本文讨论的是项目中承载业务语义的自然语言部分。判断一段文本是否适用本文原则，关键看它是否在帮助项目理解概念、说明规则、确认口径和记录取舍。它可以出现在文档、术语表、测试说明或代码注释中；载体不同，语言原则相同。

实现符号按工程约束处理。程序、数据、接口和第三方系统命名，可以遵循相应生态与团队惯例。关键要求是同一业务概念在整个系统中保持稳定对应，避免自然语言说明和实现别名之间分散成无法匹配的多套说法。

主要服务技术维护的说明，可以按工程协作习惯处理。它们关注工程操作、框架限制、构建流程和实现细节，语言选择由团队工程实践决定。

## 反对意见

**各项目都使用自己的工作语言，会不会降低跨项目复用？**

跨项目复用重点在语义迁移。LLM 可以读取源项目中的领域知识、规则模式和术语关系，再结合目标项目的术语表、已有项目真相、代码结构和工作语言，生成符合目标项目表达习惯的版本。

这样迁移后的内容能够像为目标项目原生编写的一样参与检索、审查和持续维护。能够迁移的是领域知识、规则模式、建模经验、术语关系和历史决策的启发；只有被目标项目确认并采用的内容，才会成为目标项目真相。

**自然语言与实现命名不同，会不会制造混乱？**

实现命名和自然语言命名表达项目真相的不同部分。实现命名表达程序结构，受组件划分、重构、框架惯例、数据库设计和符号系统影响；自然语言命名表达业务语义，需要跟随工作语言保持稳定。

项目重构后，实现名可能变化，业务概念仍然可以保持稳定。二者通过术语表、引用关系和上下文建立对应，能够同时支持工程定位和业务理解。

## 结论

项目自然语言部分应与工作语言一致。工作语言承载项目实际处理业务语义的方式，也能被主要维护者持续复用。

这条原则让项目真相中的自然语言内容、术语表、业务规则、验收口径、决策记录和业务含义注释保持同一语义空间，减少转译损耗和术语漂移，提高检索、抽取和投影视图生成效率。实现命名可以按工程约束演进，自然语言部分则应稳定承载业务语义。
