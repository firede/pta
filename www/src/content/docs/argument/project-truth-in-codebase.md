在 coding agent 参与长期迭代的软件项目中，项目真相必须进入代码库，并与实现共享版本边界。

本文沿用 [什么是项目真相](/argument/what-is-project-truth/) 的定义：项目真相是当前项目实际依照的内容。其中一部分存在于代码实现中，另一部分由自然语言、术语表、决策记录和必要的历史记忆补足。

已经由代码表达的部分本就在代码库中。本文所说的“进入代码库”，重点指需要由自然语言、术语表、决策记录和必要的历史记忆补足的那部分项目真相；它们也要与代码实现处在同一个版本化空间里。

## 背景

传统产研协作把需求和实现分开。产品通过 PRD、Spec、Issue 等形式交付需求，研发再把这些描述转化为代码。这套模式主要服务于人与人的异步协作：不同角色、不同工具、不同交付边界，需要一份可以转交、评审和排期的材料。

这种分离来自组织协作结构，不代表需求材料和实现必须长期分属两个版本空间。在 coding agent 参与实现之后，需求澄清和代码变更的距离被显著缩短。很多时候，只要需求被明确定义，约束被澄清，上下文被补足，实现就可以很快生成。

需求材料的价值在于约束和指导实现。长期迭代项目需要把当前项目实际依照的内容保存在实现所在的版本系统中，避免它们滞留在外部文档、人的记忆、会议记录或过期 issue 里。

## 理由

代码没有表达、难以表达或容易被误读的项目真相停留在代码库之外，会制造持续的转译成本。

传统 PRD、Spec、Issue 往往先从已有项目状态中抽取上下文，再加入背景、功能描述、变更说明和验收标准。这一步会把代码实现、产品现状和业务判断转译成文档。研发或 agent 接收到这些材料后，还要把文档重新映射回代码库，判断它对应哪些模型、接口、页面、流程、边界条件和测试。于是，同一份项目真相经历了从代码到文档、再从文档回到代码的往返转译。

每一次转译都会带来信息损失、转述偏差、版本不同步和上下文缺失。在人类协作主导的模式下，这种成本可以被组织流程吸收。到了 agent 参与实现时，这些转译会持续阻塞代码变更。agent 主要在代码库中工作，实际变更最终表现为代码 diff，并通过测试、构建和部署验证。代码没有表达、难以表达或容易被误读的项目真相一旦停留在另一个版本空间，agent 每次改动都要依赖用户转述、临时检索或额外拼接上下文。

项目真相进入代码库后，这段跨版本空间的往返转译会被消除。代码实现继续表达项目真相中可以运行的部分；自然语言、术语表、决策记录和必要的历史记忆，在同一个版本边界内补足代码没有表达、难以表达或容易被误读的部分。两部分一起维护后，变更可以直接发生在实现所在的版本系统中，而不必先生成一份外部文档，再把文档翻译回代码变更。

外部知识库仍然有价值，适合阅读、讨论、展示和收集材料。它的弱点在于版本边界：它不与代码共享提交历史、审查流程、回滚边界和变更上下文。对于长期迭代项目来说，当前项目实际依照的内容如果不随实现同步演进，就会逐渐变成不可靠的旁路信息。

## 边界

进入代码库的是已经成为项目真相的内容；产生这些内容的过程，可以留在各自合适的工作流中。

团队会议、用户反馈、产品讨论、与 coding agent 的多轮澄清、基于反馈自动触发的 pull request，都可以产生项目真相。这些过程本身不必原样进入代码库。只有那些经过澄清、压缩和确认，并成为当前项目实际依照的内容，才需要沉淀为当前版本的业务状态、领域约束、真实世界条件、商业约束、产品语义、稳定术语、决策结果和关键变化原因。

这套模式主要适用于需要长期迭代的软件项目。长期迭代需要稳固、可追溯、可复用的当前依据。单次原型探索的目标是快速做出来、快速看见、快速反馈。此时项目真相可以暂时存在于对话上下文、临时材料和即时判断中；过早沉淀进代码库，反而会拖慢探索速度。

项目真相进入代码库，是底层结构判断；上层如何访问、审计、修改和发布这些真相，属于应用层设计问题。产品、设计、业务和研发不必都直接面对 Git 或代码编辑器。GUI、对话式 agent、需求系统或自动化工作流，都可以作为审计、核查和更新项目真相的入口。

决定项目真相的位置时，应先看产品如何被定义、实现、验证和迭代，再看不同人员、agent 和工具应该通过哪些入口创造价值。

## 反对意见

**代码库会不会变成产品材料垃圾场？**

这个风险来自入库标准失守。若是把所有与产品有关的材料都放进仓库，代码库必然会失控。

本文采用更窄的入库标准：进入代码库的内容必须已经成为当前项目实际依照的内容。用户研究、市场调研、商业分析可以作为输入留在独立项目中。只有当这些材料的结论已经转化为产品行为、权限规则、领域模型、真实世界限制、商业约束、部署限制、稳定术语或方案取舍时，才需要随实现版本化。

**PRD、Spec、Issue 是否因此被否定？**

不会。PRD、Spec、Issue 的问题不在名字或形式，而在传统用法把它们放在需求与实现之间，承担最终依据的角色。

在 agent 时代，它们可以作为上层封装继续存在：讨论入口、任务编排、审计界面、变更申请或历史记录。要进入代码库中的项目真相，则需要经过澄清、压缩、确认和转化。

## 结论

在 coding agent 参与长期迭代的项目中，不能继续把代码没有表达、难以表达或容易被误读的项目真相放在实现所在的版本系统之外。需求和实现本来服务于同一个目标；coding agent 缩短了澄清到变更的距离，也让传统分离的成本更明显。

项目真相必须进入代码库，因为当前项目实际依照的内容需要与实现共享同一个版本边界。这样，人、agent 和工具才能围绕同一份当前依据工作，减少在外部材料和代码变更之间来回搬运上下文。

本文只确定位置与版本关系，不展开项目真相进入代码库后的具体形式、审计流程、更新机制，以及它与测试、类型、配置、文档和 agent 工作流的协同方式。
