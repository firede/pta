## 主张

在使用 coding agent 长期迭代的软件项目中，项目真相必须进入代码库，并与实现共享版本边界。

项目真相指当前版本中已经确认、并对实现产生约束的事实：业务状态、领域逻辑、商业限制、产品语义、关键决策及其结果。代码实现表达了一部分产品运行真相；另一些约束无法仅靠代码自证，例如物理资源限制、历史方案包袱、业务编码规则、关键取舍理由。它们同样影响实现判断，也应该被当前版本保存和审查。

## 背景

传统产研协作把需求和实现分开。产品通过 PRD、Spec、Issue 等形式交付需求，研发再把这些描述转化为代码。这套模式首先服务于人与人的异步协作：不同角色、不同工具、不同交付边界，需要一份可以转交、评审和排期的材料。

这种分离来自组织协作结构，并不意味着需求和实现必须长期分属两个事实源。在 coding agent 参与实现之后，需求澄清和代码变更的距离被显著缩短。很多时候，只要需求被明确定义，约束被澄清，上下文被补足，实现就几乎同时可以生成。

需求的目的始终是约束和指导实现。长期迭代项目需要把这些约束保存在实现所在的版本系统中，避免它们滞留在外部文档、人的记忆、会议记录或过期 issue 里。

## 理由

项目真相和实现分离，会制造持续的信息交换成本。

传统 PRD、Spec、Issue 往往先从已有产品状态中抽取上下文，再加入背景、功能描述、变更说明和验收标准。研发或 agent 接收到这些材料后，还要把它们重新映射到代码库中。这个过程经历多次转换：产品真相到需求文本，需求文本到实现理解，实现理解到代码变更。每次转换都会带来信息损失、转述偏差、版本不同步和上下文缺失。

在人类协作主导的模式下，这种成本可以被组织流程吸收。到了 agent 参与实现的场景，它会变成系统性瓶颈。agent 的主要工作界面是代码库，实际变更最终表现为代码 diff，并通过测试、构建和部署验证。约束实现的事实一旦停留在另一个版本空间，agent 每次改动都要依赖用户转述、临时检索或额外拼接上下文。

可以把项目真相看作项目的记忆，把代码实现看作项目的执行。记忆和执行分处两个系统，每次实现都需要跨系统搬运上下文，就会出现类似带宽瓶颈的问题：延迟高、容量低、容易丢失信息，也难以确认当前实现对应的是哪一版事实。项目真相进入代码库，是为了让记忆和执行处于同一个版本化空间中。

外部知识库仍然有价值，适合阅读、讨论、展示和收集材料。它的弱点在于版本边界：它不与代码共享提交历史、审查流程、回滚边界和变更上下文。对于长期迭代项目来说，约束实现的事实如果不随实现同步演进，就会逐渐变成不可靠的旁路信息。

## 边界

入库的是项目真相的结果；产生这些结果的过程，可以留在各自合适的工作流中。

团队会议、用户反馈、产品讨论、与 coding agent 的多轮澄清、基于反馈自动触发的 pull request，都可以产生项目真相。这些过程本身不必原样进入代码库。只有那些经过澄清、压缩和确认，并开始约束当前实现的结论，才需要沉淀为当前版本的业务状态、领域约束、产品语义和业务决策快照。

这套模式主要适用于需要长期迭代的软件项目。长期迭代需要稳固、可追溯、可复用的事实基础。单次原型探索则不同，它的目标是快速做出来、快速看见、快速反馈。此时项目真相可以暂时存在于对话上下文、临时材料和即时判断中；过早沉淀进代码库，反而会拖慢探索速度。

项目真相进入代码库，是底层结构判断；上层如何访问、审计、修改和发布这些真相，属于应用层设计问题。产品、设计、业务和研发不必都直接面对 Git 或代码编辑器。GUI、对话式 agent、需求系统或自动化工作流，都可以成为审计、核查和更新项目真相的界面。

职业角色也不应先验决定协作架构。更合理的顺序是：先看产品如何被定义、实现、验证和迭代，再看不同人员、agent 和工具应该在哪里创造价值。

## 反对意见

**代码库会不会变成产品材料垃圾场？**

这个风险来自入库标准失守。若是把所有与产品有关的材料都放进仓库，代码库必然会失控。

本文主张的入库标准更窄：入库材料必须已经约束当前实现，并沉淀为当前版本的业务事实。用户研究、市场调研、商业分析可以作为输入留在独立项目中。只有当这些材料的结论已经转化为产品行为、权限规则、领域模型、部署限制或方案取舍时，才需要随实现版本化。

**PRD、Spec、Issue 是否因此被否定？**

不会。PRD、Spec、Issue 的问题不在名字或形式，而在传统用法把它们放在需求与实现之间，承担最终事实源的角色。

在 agent 时代，它们会被重构为上层封装：讨论入口、任务编排、审计界面、变更申请或历史记录。它们可以继续存在；要进入代码库中的项目真相，则需要经过澄清、压缩、确认和转化。

## 结论

在 agent 时代，长期迭代项目不能继续把项目真相放在实现之外。需求和实现本来服务于同一个目标；coding agent 缩短了澄清到变更的距离，也让传统分离的成本更明显。

项目真相必须进入代码库，因为约束实现的事实需要与实现共享同一个版本化事实层。这样，人、agent 和工具才能围绕同一份当前事实工作，减少在外部材料和代码变更之间来回搬运上下文。

本文只确定位置与版本关系。项目真相进入代码库后，应该以什么形式存在，如何被审计，如何被更新，如何与测试、类型、配置、文档和 agent 工作流协同，应交给其他立论分别承担。
