项目真相，是当前项目实际依照的内容。它决定项目现在如何运行、如何被理解、后续修改和判断时必须参考什么，也为不同任务场景生成投影视图提供依据。

项目真相主要服务于项目的持续迭代。只要项目会被继续理解、修改、解释、审计，或被用来生成投影视图，就需要一份当前依据。使用这份依据的可以是人，也可以是 agent；项目真相面向的是持续变化中的理解和判断。

## 背景

使用项目真相这个说法，是为了强调当前有效。它指当前版本中项目实际依照的核心内容；这些内容可以随版本变化，范围由项目采用和当前有效限定。

文档、上下文、事实分别指向不同问题：文档强调表达形式；上下文通常和场景绑定，更像某次任务或某个视角下的切面；事实是关于领域或真实世界的客观陈述。项目真相关心的是项目从这些材料中实际采用了什么，并按什么成立、运行和演进。

因此，定义项目真相时，核心问题是：当前项目实际依照什么？哪些内容一旦变化，会影响项目的运行、理解、维护、判断或投影视图生成？

## 理由

**项目真相的一部分，就存在于代码实现中。**

当前实现已经表达了产品实际怎样运行、数据如何组织、规则如何落地、边界如何处理。代码已经清楚表达的真相，保留在代码里即可，不要再用文档复写同一份真相。复写会制造同步成本，也会增加理解难度。

**自然语言更适合补足代码没有表达、难以表达，或容易被误读的部分。**

这些内容包括代码无法充分表达的领域知识、隐性知识、真实世界知识、业务规则、商业约束和实际生效的产品权衡。例如，领域中确实存在某些事实，但项目真正需要记录的是自己如何采用这些事实，并如何把它们落到产品行为和实现方式里。

“人有恒牙和乳牙”是领域事实；“本系统采用 FDI 编号记录牙齿，早期未区分恒牙乳牙，现在统一用恒牙编号记录，并增加布尔变量区分恒牙乳牙”才是项目真相。前者是领域知识，后者是当前系统采用的记录规则和历史取舍。

**稳定术语表也属于项目真相。**

项目需要在同一套词汇中被理解和修改。术语表的作用是消除模糊空间，减少同义词、近义词、行业俗称和错误写法带来的歧义。在 agent coding 中，这一点尤其重要：agent 会根据词汇建立上下文关联。如果项目没有稳定术语表，就容易混用“账号”和“账户”、“用户”和“客户”，甚至把“正颌”和“正颚”这类专业词弄错。

统一词汇的价值在于让人和 agent 在同一个语义空间里工作。

**项目真相还需要保留少量历史记忆。**

这里的历史用于记录关键变化原因，完整讨论过程可以留在提交记录、issue 或外部材料里。平时阅读项目时，可以先看当前仍然生效的决策、规则和权衡；这些内容应该直接体现在当前项目真相里。

历史的作用是在需要追问“为什么曾经改成这样”时，提供低成本的时间维度线索。Git 通常能回答改了什么、什么时候改的、谁改的；项目真相中的历史记录主要回答为什么这样改。因此，它适合记录领域策略、业务逻辑、产品权衡、对外交付语义、重要术语等关键变化的原因。

这类历史记录要谨慎写入，也要谨慎处理。写入之后，删除或压缩需要明确项目判断；单个 agent 当下看不出用途，不足以成为清理理由。未来的追溯、审计、投影生成或对外交付解释，可能正需要这些变化原因。

## 边界

**领域事实可以成为项目真相的来源。**

这里说的事实，指关于领域或真实世界的客观陈述。一个事实只有被当前项目采用，并且影响产品行为、业务解释、实现判断、术语使用、对外交付或后续维护时，才成为项目真相。

**多种载体都可以表达项目真相。**

代码、测试、配置、自然语言文档、术语表和决策记录，都可以表达项目真相。代码已经清楚表达的部分，可以直接保留在代码里；文档主要补足代码没有表达、难以表达或不适合承担的部分。

**项目真相的范围由项目需要划定。**

开发者、产品经理、设计师、领域专家或业务方掌握的内容，都可能贡献项目真相。真正进入项目真相的，是当前项目需要被持续复用的部分，范围可能跨越实现、领域、业务、商业、产品和真实世界限制。

**投影视图是项目真相的消费结果。**

投影视图面向某次任务、某个 agent 执行场景、某次审计、业务理解、产品调研或管理查看，从项目真相中抽取、重组和呈现一部分内容。它可以被生成、展示、缓存、传递给 agent 使用。只要能够基于项目真相语义复现，通常保持为生成结果或缓存结果即可。

本文只定义项目真相是什么。项目真相如何进入版本库，如何审查、更新、废弃和治理，不在本文展开。

## 反对意见

**项目真相会不会变成什么都往里放？**

这个担心成立。项目真相一旦缺少治理，就会退化成材料堆积。因此，项目真相必须有治理机制：进入有标准，变更有审查，并且随版本库一起维护。

进入项目真相的内容，应当是当前项目实际依照的内容。当前版本已经停止采用的内容，应当随项目变更一起修改并提交，避免继续混在当前真相中。

**既然项目真相会随版本变化，为什么还叫真相？**

这里的真相强调当前有效，并允许随版本变化。

某个业务规则、术语、实现方式或产品权衡，可能曾经是项目真相，也可能在新版本中被替换。这种变化符合软件项目的版本化存在方式。

对任一版本来说，项目实际依照什么，人和 agent 就应该按什么理解、修改和判断项目。项目真相的意义，正在于明确这个当前版本的有效核心。

## 结论

项目真相，是当前项目实际依照的内容。一部分由代码实现表达，另一部分由自然语言、术语表、决策记录和必要的历史记忆补足。

它关注的是：当前项目如何运行，如何被理解，后续修改时必须尊重什么，以及不同任务场景下的投影视图可以从哪里生成。事实材料可以作为输入，投影视图可以作为输出；只有被当前项目采用并持续影响判断的内容，才需要沉淀为项目真相。

先说明项目真相是什么，项目真相架构才有明确的基础。
