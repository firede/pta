领域声明定义领域如何在版本库中被界定、标记和连接。

> 本规范落实 [项目真相应按领域组织](/argument/project-truth-by-domain/) 的组织原则，并为 [新鲜度治理](/argument/project-truth-freshness-governance/) 的核查提供机器可判定的领域边界。立论回答为什么，本规范回答怎么做。

## 约定用语

本规范用以下用语表达要求等级：**必须**，实现没有例外；**应当**，默认遵循，偏离需要有可说明的理由；**可以**，由项目自行决定。否定形式与等级对应：**不得** 是必须级的禁止，**不应** 是应当级的禁止。

## 领域标记

一个目录因携带 `TRUTH.md` 而构成领域。这是领域存在的唯一判定：人和机器依据同一标记识别领域，不依赖约定推断。

`TRUTH.md` 承载该领域的真相记录：概念、规则、约束、取舍及其原因。复述实现细节的内容不写入。

领域以其主张的目录路径为标识。目录内 `TRUTH.md` 的位置即领域边界，**不得** 在 frontmatter 中声明路径。领域不声明名称或描述，供人阅读的显示名由消费界面按需导出。frontmatter 全部可选：

```markdown
---
dependsOn:
  - path: src/components/screening-form
    reason: 展示规则依赖检查表单的录入口径
---

……
```

frontmatter 的语法遵循 YAML 1.2；各字段的形态以本规范相应小节的定义为准，未识别的字段 **应当** 忽略。无法按 YAML 1.2 解析的 frontmatter 构成核查信号。

**`dependsOn`（可选）** 声明无法从已有结构推导的记录依赖：本领域的真相记录依赖所指领域的内容，对方变更时，本领域进入核查候选。每项包含 `path` 与 `reason`，以本领域视角说明依赖对方什么。

每项的 `path` **必须** 是版本库中某个领域的标识，指向不存在领域的依赖构成违例：依赖的全部作用是让对方的变更把本领域送入核查候选，悬空的目标使这条传播静默失效。

依赖是有向的，**必须** 声明在被影响的一方；反向关系（谁依赖本领域）由工具扫描全部领域声明推导，**不得** 在被依赖方镜像声明。发现未声明的依赖时，无论发现者是谁，声明都补写到被影响的领域。

能从实现依赖、记录引用或目录层级推导的关系，**不应** 写入 `dependsOn`：声明文件本身是真相记录，同样受新鲜度治理，字段越少，需要维持新鲜的面越小。

## 领域层级

领域的上级，是目录树中最近的携带 `TRUTH.md` 的祖先目录所构成的领域。外置声明的领域以其声明的 `path` 计算领域层级；携带 `files` 的外置领域，其指向的目录自身构成领域时，以该领域为上级。层级由目录结构给出，`TRUTH.md` **不得** 另行声明上下级关系。

上级领域的记录构成下级领域的背景。下级 `TRUTH.md` 不复写上级已经成立的内容，只记录本领域更具体的判断。

仓库根目录携带 `TRUTH.md` 时，构成根领域，承载项目级判断，其余领域最终在其背景内成立；未携带时，顶层领域没有上级领域。

## 伴随文件

**`GLOSSARY.md`（可选）** 维护该领域的术语。术语沿层级共享背景：下级领域 **不得** 与上级术语表发生冲突性重定义，冲突构成核查信号。`GLOSSARY.md` 可以由 LLM 起草和维护，采用与否由维护者裁决；它是项目真相的一部分，不是投影视图。

**`RESIDUE.md`（可选）** 记录该领域仍在生效的残留：过去版本作用于真实世界留下的后果，在当前实现没有锚点，但仍影响后续判断。每条 **应当** 简短，并附时间或版本线索，为查看变更记录提供入口。入选判据是仍然影响后续判断；不再影响的条目 **应当** 删除。`RESIDUE.md` 不是版本发布记录，也不是过程性材料的存放处；解释现行判断的历史原因随判断记录在 `TRUTH.md`，**不得** 写入残留。

**`PENDING.md`（可选）** 收纳该领域的待裁决条目。为项目工作的参与者撞到真相记录未钉、合理实践给不出唯一解、而项目可能在乎的选择，且自己无权裁决时，把它连同当前采取的处置记录于此，移交维护者裁决；有权裁决的发现不经过此文件，直接走正常变更流程。条目主张的是仍在等待裁决，裁决落地后条目随之失实，**必须** 随处置删除：判断入选的，新条目随同一变更进入 `TRUTH.md`；确认放权的，确认由裁决缓存吸收。`PENDING.md` 不承载项目真相，条目对后续迭代没有约束力，其中记录的处置只是现状。

## 外置领域声明

领域无法以单一目录显现，或不能在目标目录中放置文件时，**可以** 使用外置领域声明。

外置声明收纳于外置声明根中，按领域名组织为单层目录，**不得** 嵌套；仓库根目录的 `.pta/` 是默认的外置声明根，其他根由 `pta.toml` 声明。外置声明根中的 `TRUTH.md` 只按外置声明解释，不使其所在目录构成领域。每个外置领域的入口是 `{externalRoot}/{name}/TRUTH.md`，frontmatter 中 `path` **必须** 填写并指向真实目录；`files` 可选，仅在需要把领域限定为目录内部分文件时使用：

```markdown
---
path: internal/dsl
files:
  - lexer.go
  - parser.go
  - lexer_test.go
  - parser_test.go
---

……
```

仅 `path` 的外置领域以其主张的目录路径为标识，与目录内声明一致；携带 `files` 的外置领域以声明所在的目录路径（`{externalRoot}/{name}`）为标识。

`files` 的成员是相对 `path` 的路径片段，沿用标识规范的路径段规则：以 `/` 分隔，段 **不得** 为空或为 `.`、`..`，拼写逐字节一致、不折叠大小写。成员与 `path` 拼接所得的仓库根相对路径 **必须** 指向版本库中真实存在的文件而非目录；列表内 **不得** 重复。

`files` 是外置声明的表达上限：文件列表应保持少量、明确、可枚举；这样界定的领域作为层级末端存在，不提供下级空间。伴随文件与目录内领域相同，放置在 `{externalRoot}/{name}/` 下。

外置声明受以下约束：

- 主张整个目录的声明，无论是目录内 `TRUTH.md` 还是仅 `path` 的外置声明，每个目录至多一个，重复主张构成冲突，冲突是核查信号。
- 携带 `files` 的外置声明只主张部分文件，**可以** 与整目录声明共存，构成其下级；多个共存时，文件列表 **不得** 重叠，重叠同样构成冲突。
- 携带 `files` 的声明，成员 **不得** 落入 `path` 之下更具体领域的边界内，越界构成冲突：文件子集是层级末端，不跨越更具体的领域。
- 外置声明 **应当** 作为过渡或例外存在：结构能够重排时，优先让领域回到目录内声明；结构无法重排时，例外 **可以** 长期存在。

## 多包仓库

单一 `.pta/` 不足以服务多包仓库时，**可以** 在仓库根目录放置 `pta.toml`，通过 `externalRoots` 声明多个外置声明根：

```toml
externalRoots = ["packages/web/.pta", "packages/client/.pta"]
```

本规范只保留该扩展点，字段与解析规则由集成规范定义。

## 与投影视图的关系

领域声明及其伴随文件是项目真相的记录，**必须** 进入版本库。基于领域生成的投影视图是消费结果，**不得** 进入版本库；其生成与缓存位置由编译规范定义。

## 范围之外

`TRUTH.md` 与伴随文件正文的内容结构（条目形态、写法约定）属于内容结构规范；核查节点如何消费领域声明与 `dependsOn` 关系、待裁决条目如何作为信号进入回路，属于治理规范；领域声明与 AGENTS.md 等既有工程文件的关系，属于集成规范。
