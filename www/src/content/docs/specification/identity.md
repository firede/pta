标识定义内容如何获得可跨工具复算的身份。治理的缓存键、编译的来源标识与条目身份都由这里的部件构成；两个实现对同一内容必须算出同一标识，共享与流转才能成立。

> 本规范落实 [可推导内容应留在工具层](/argument/derivable-content-in-tool-layer/) 的内容哈希键法，版本标识建立在 [依据部分必须随执行部分版本化](/argument/grounding-part-versioned-with-execution-part/) 提供的共享版本边界上。立论回答为什么，本规范回答怎么做。

## 约定用语

本规范用以下用语表达要求等级：**必须**，实现没有例外；**应当**，默认遵循，偏离需要有可说明的理由；**可以**，由项目自行决定。

## 条目标识

条目以其内容的哈希为身份。内容结构规范约束条目占据源码中的单一行，取哈希的文本即条目标记之后至行尾的部分，**必须** 按以下规则规范化：

- 去除首尾空白，内部空白原样保留；
- 不做转义与语法解析，Markdown 语法字符按源码原样属于内容；
- Unicode 按 NFC 规范化。

加粗的术语名变化即身份变化，这是正确行为。

规范化后的文本以 UTF-8 编码，哈希算法为 SHA-256，标识为其十六进制小写表示。截断显示属于工具层，不影响存储的完整标识。

## 文件与版本标识

文件以其字节内容的 SHA-256 为标识，不做规范化：文件是版本库的原生对象，字节即身份。

已入库状态以版本库的原生版本标识表达，如 git 的提交哈希，本规范不另造版本编号。含未入库变更的状态，以基版本加变更记录的集合表达：每项记录包含路径、变更类型——新增、修改或删除——文件模式与变更后内容的哈希，删除项不含内容哈希，记录按规范路径排序。仅凭内容哈希的集合不足以定身份：文件互换内容、删除与改名都不改变哈希集合，却改变项目语义。集合的序列化属于集成规范，标识的语义见编译规范的可追溯义务。

## 领域标识

领域以路径为标识，标识的判定规则由领域声明规范给出，本规范定义路径的书写形态：相对仓库根，以 `/` 分隔，不含前导 `./` 与尾随 `/`；路径段 **不得** 为空，**不得** 为 `.` 或 `..`。拼写 **必须** 与版本库树中的条目逐字节一致，不随宿主文件系统折叠大小写，也不做 Unicode 归一。仓库根构成的领域以空字符串为路径。

## 部件类型

进入键与来源标识的部件分四类，类型随部件参与组合：

- **条目**：按条目标识计算。
- **文本**：不落在版本库中的自由文本，如临时提示词与工具内建场景的表达。规范化为换行统一 LF、去除首尾空白、Unicode 按 NFC，以 UTF-8 编码取 SHA-256。
- **文件**：按文件标识计算。
- **原生**：版本库事件、发布版本号、转换实现的名称与版本等自带标识的对象，用其原生标识，不另行哈希。

由多个部件构成的集合——文件集、证据组，成员各自按类型计算；集合的排序、边界与序列化属于集成规范。

## 范围之外

多个部件如何组合并序列化为一个键，属于集成规范。哈希算法与规范化规则的演进随本规范版本化，实现如何声明所遵循的版本，属于将来的一致性约定。
