为受托任务分支的合并门禁准备审核材料。裁决与合并始终归人；本技能只备料、答问、按指示执行。

前提：

- 确认目标分支；基线取其与主线的 merge-base。
- 工作树干净；归因与提交权限遵循 AGENTS.md「提交与归因」。
- 材料从仓库记录（提交、真相文件、行为日志）产出，不依赖原始会话在场——冷启动的 agent 也能完整执行本技能。

工作方式（四个审核面按序产出）：

1. 判断层——真相变化：对基线到分支头，输出真相载体（TRUTH.md、GLOSSARY.md、RESIDUE.md、PENDING.md、领域声明）的条目级增删，按领域归组；在分支上运行 `pnpm pta check`、`pta pending list`、`pta inspect list`，汇报遗留的未处置信号、待裁决条目与到期线索；运行 `pnpm pta changes <基线>`，逐域核对「实现被触而真相记录未动」的漂移是否已由待裁决条目、真相变更或简报解释——解释不了的漂移是记录缺口，先补记录再继续。（`pta diff` 落地后改用之。）
2. 叙事层——简报：按提交梳理做了什么与关键取舍；被驳回的评审意见附理由，并给出每条理由在耐久记录（commit message、真相或待裁决条目）中的出处——说得出但找不到出处的理由是记录缺口，先补记录再继续。
3. 行为层——试驾清单：按变更的可运行表面裁剪，列出值得亲手验证的场景，附可直接执行的命令与预期表现；纯文档等无运行表面的变更，明说「无试驾场景」即可，不硬凑，审核由判断层与叙事层承担。
4. 实现层——交叉评审：用户指定评审 agent 时，评审在其自有的持续会话中进行——多轮发现与复核需要热上下文；发现回流后逐条核实并裁决，采纳即修复并验证，驳回附理由，结果并入简报。

合并执行（仅在用户裁决通过后）：

- 由用户合并，或经用户明确指示代为执行；merge commit 携带简报的裁决摘要，author 为用户。
- 合并后提醒清理：任务分支与备份分支的去留由用户定。

完成条件：

- 四个审核面材料齐备，且未看过原始会话的人或 agent 仅凭材料与仓库记录足以重建决策。
- 用户完成裁决：合并、退回修改或搁置。

输出规范：

- 审核材料在会话中以散文与清单呈现，不写入版本库。
- 不自主发起合并；退回修改时，修复提交仍按受托模式署名。
- 用户就分支提问时直接作答，必要时现场运行代码验证，不引导用户去其他界面。
