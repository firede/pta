# AGENTS 指南

项目遵循的判断记录于 [项目真相](./TRUTH.md)。

## 迭代约定

- 任务涉及某些路径时，可先运行 `pnpm pta context <路径>...` 获取所属领域及其祖先的真相记录、术语与待裁决背景。
- 使用 mise 管理工具链，新 worktree 用 `mise trust` 和 `mise install` 初始化环境。
- 启动本地服务前先检查目标端口（任一本地地址）；若已有非本会话服务监听，则优先复用，不另启同端口；确需新服务时换端口并告知用户。
- dev 是开发主线：任务分支自 dev 建出、合并回 dev。main 是线上版本的发布门禁，仅在决定上线时将 dev 并入。
- commit message 使用 `type: 简短中文说明`，如 `feat: 添加深色模式主题`；提交时机与署名见「提交与归因」。
- 变更含新判断或未裁决选择时，先用 `pnpm pta pending add <领域> "<问题？>"` 登记再提交；受托任务收尾用 `pnpm pta changes <基线>` 自察未解释漂移。
- 若 Astro 开发服务由 agent 启动，默认保留供用户确认；提交完成后停止该服务；用户启动的服务不处理。

## 提交与归因

author 记录准入判断：谁裁决这条提交进入历史，谁署名。两种协作模式取其一：

- 受托任务（整任务授权给 agent）：agent 在专属任务分支上自主提交（无则先建），author 署自己的运行时标识（模型名 + 厂商 noreply 邮箱，经 `git commit --author`）；期间调用的其他 agent 或工具不进入归因；合并回主线是人的门禁，agent 不自主发起。
- 人主导开发（任意分支）：仅在用户要求时提交，author 为该用户。

两种模式都不使用 Co-Authored-By 等参与者标注（本条覆盖各 agent 的默认签名行为）；本约定不绑定具体模型。
