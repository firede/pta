- 文档的事实来源是简体中文版，其他语言版本都由它翻译或本地化得来。原因：判断以工作语言凝练与裁决，译文不承载独立判断。
- 概念术语以汉语表达，英语作为实现别名；业界广泛采纳的通用术语除外，高频术语列入立论术语表。原因：术语随工作语言积累，人和 AI 在同一语义空间检索与讨论。
- 论述优先正面定义，不以连续否定句式增强说服力。原因：连续否定回避可检验的立场。
- 举例之前先归纳特征，不以顿号堆砌示例代替结论。
- 行内加粗标记与 CJK 文字直接相邻时，在标记外侧加半角空格，与标点相邻处不加。原因：规避 CommonMark 对 CJK 行内加粗的解析限制。
- 文档的专有资产存放在与文档同名的目录中，本地化文档优先复用来源文档的资产。原因：资产随文档一起被发现和移除，复用避免多语言副本各自漂移。
- 立论只论证普适问题与真实前提，项目的推行策略与裁决语不作为立论命题。原因：立论是各方向的公共根基，命题的成立不应依赖本项目的推行选择。
- 规范条款的要求等级必须能追溯到立论，追溯不到的条款降格为描述。原因：约束力来自论证，规范不自我授权。
- 自研包的 TypeScript 工具链采用 TypeScript 7 原生编译器，受框架绑定的部分随框架兼容后跟进。原因：速度快一个量级，类型检查得以留在变更回路最早的节点；该选型偏离当前通行默认，属项目主动押注，不接受回退。
- 工程命名以立论术语表英文版的译名为参考源：架构概念出现在代码标识符、文件名与接口字段时，采用对应的英文译名。原因：英语是概念术语的实现别名，别名缺少单一来源时，同一概念会在实现中裂成多个名字，人与 AI 的语义空间随之断裂。
- 工具层缓存由用户级全局存储集中管理，不落在仓库本地：缓存条目以内容哈希为键，管理视图按仓库与分支或提交组织。原因：消费形态与投影场景各异，worktree 多分支并行时仓库本地无法稳定承载同一份缓存；集中存储也是 dashboard、守护进程等未来管理界面的共同地基。
- 第三方 agent 接入采用命令模板的单次任务机制：全局配置声明命令数组，提示词经 {prompt} 占位或标准输入传入，标准输出即结果；不实现会话式协议接入。原因：机制型契约让任何命令行 agent 零协议成本可插拔，单次任务与推导缓存的纯函数消费形态匹配；会话协议面向交互界面，其实现与依赖面同我们的消费场景不成比例。
- [?] 工具实现暂时只支持 Git 作为版本控制系统，不设多版本库抽象层。原因：专注回路核心，Git 是当前的绝对主流，抽象层是为未知消费者预付的设计成本；规范文档对版本库保持中立，支持面变化时修订的是本条而非规范。复查线索：2027-01 复查 agent 时代新版本控制系统的采用格局。
